    数据库使用规范与最佳实践

    MySQL 最佳实践

    性能优化黄金法则

    1. 合理选择存储引擎

    • InnoDB 是 MySQL 8.0+ 的默认事务性存储引擎，支持 ACID 事务、行级锁定、外键约束等功能
    • 在需要事务支持、并发写入和外键约束时，应使用 InnoDB 存储引擎
    • MEMORY 存储引擎可用于临时性高速访问场景，但需注意数据不持久化
    • 对于日志或其他只读或极少更新的大表场景，可考虑使用 Archive 存储引擎

    2. 优化查询策略

    • 合理创建索引，关注复合索引顺序
    • 避免全表扫描和过多的 JOIN
    • 合理使用缓存机制提升性能

    3. 配置参数调优

    • 调整缓冲池大小(buffer pool size)
    • 优化连接数和线程配置
    • 合理设置日志参数平衡性能与安全

    安全配置要点

    1. 权限访问控制

    • 遵循最小权限原则分配用户权限
    • 定期审查和清理不必要的用户账号
    • 合理设置 IP 限制和网络访问策略

    2. 数据保护策略

    • 启用审计日志监控重要操作
    • 配置 SSL/TLS 加密传输数据
    • 考虑透明数据加密保护静态数据

    MySQL 命名规范

    1. 对象命名通用原则

    • 使用小写字母和下划线组合，避免驼峰命名法
    • 表名不超过64个字符
    • 避免使用MySQL保留字作为标识符

    2. 数据库对象命名规范

    • 表名：全部小写，多个单词使用下划线分隔，如 user_info
    • 字段名：采用有意义的英文词汇，如 user_name, age
    • 索引：PRIMARY KEY 使用 pk_表名，普通索引使用 idx_表名_字段名
    • 外键：使用 fk_表名_关联字段名 格式命名
    • 视图：以 v_ 作为前缀，如 v_user_detail

    3. 特殊字段命名

    • 主键：统一使用 id (对于关联外键使用表名_id 的形式)
    • 时间戳字段：created_time, updated_time 或 create_time, modify_time
    • 布尔值字段：使用 is_, has_, can_ 等前缀，如 is_valid, has_permission
    • 软删除标记：del_flag, is_deleted 等

    逻辑设计规范

    1. 规范化与反规范化平衡

    • 一般遵循第三范式，减少数据冗余
    • 对于高频查询字段，可适度反规范化以提高性能
    • 对大表进行垂直拆分或水平拆分

    2. 主键设计

    • 推荐使用自增整数作为主键
    • 复合主键最多不超过2-3个字段
    • 尽量不使用 GUID 作为主键

    3. NULL 值处理

    • 尽可能使用 NOT NULL 修饰列
    • 提供合适的 DEFAULT 值避免 NULL
    • 在索引和计算中要注意 NULL 值的存在

    物理设计规范

    1. 数据类型选择

    • 根据实际需求选择最小的数据类型以节省空间
    • 时间类型统一使用 DATETIME 或 TIMESTAMP
    • 布尔值优先使用 BOOLEAN(TINYINT(1)) 类型
    • 数值类型避免精度不足和溢出问题
    • 文本字段根据长度选择 VARCHAR、TEXT 等

    2. 字符集设置

    • 统一使用 utf8mb4 编码
    • 排序规则推荐使用 utf8mb4_general_ci 或 utf8mb4_unicode_ci
    • 保持字符集与排序规则一致，避免出现比较错误

    3. 分库分表

    • 单表记录数超过1000万应考虑分表
    • 优先使用垂直切分，后考虑水平切分
    • 保持分片规则的一致性和可扩展性

    SQL 语句规范

    1. 查询语句编写

    • 禁止使用 SELECT *
    • 使用 LIMIT 优化分页查询
    • 避免在 WHERE 条件中对字段进行函数操作
    • 谨慎使用子查询，优先考虑 JOIN
    • 适当使用覆盖索引提高查询性能

    2. 更新与插入语句

    • 插入大批量数据时使用 INSERT ... VALUES ..., ... 形式
    • 使用批量更新或替换避免频繁的单条更新
    • 使用事务来保证数据一致性

    PostgreSQL 最佳实践

    性能优化黄金法则

    1. 索引设计策略

    • 理解 B-tree、Hash、GIN、GiST 不同索引适用场景
    • 优先覆盖最频繁的查询条件
    • 关注部分索引与表达式索引的应用

    2. 表分区规划

    • 合理利用范围分区、列表分区提高查询效率
    • 考虑时间维度数据冷热分离策略
    • 注意分区剪枝对性能的影响

    3. 查询编写技巧

    • 避免 SELECT * 使用，指定明确字段
    • 正确使用 JOIN 和子查询，避免笛卡尔积
    • 充分利用窗口函数完成复杂统计

    安全配置要点

    1. 用户权限管理

    • 实施最小权限原则
    • 正确配置 row-level security(RLS)
    • 区分角色类型与用途权限控制

    2. 连接认证策略

    • 合理设置 pg_hba.conf 安全认证规则
    • 配置 SSL 加密传输敏感数据
    • 考虑第三方认证后端集成方案

    PostgreSQL 命名规范

    1. 对象命名通用原则

    • 小写字母和数字，用下划线分割，如 user_info
    • 避免使用复数形式，如使用 person 表而不是 persons
    • 不要用数字开头
    • 名称长度不超过63字节
    • 避免使用SQL关键字作为对象名

    2. 数据库对象命名规范

    • 表名：用名词复数或描述业务，如 articles
    • 列名：简洁明了的名词，如 id, name, email
    • 索引：按照 idx_tablename_fieldname 规则命名，如 idx_user_name
    • 约束：fk_表名_列名, pk_表名, chk_表名_列名, uq_表名
    • 触发器：trg_tablename_event, 如 trg_user_insert
    • 函数/存储过程：func_action_description 或简写的 fn_action_desc

    3. 特殊字段命名

    • 自增主键：id (除非特殊情况，避免 user_id 等)
    • 位移标志：使用布尔值 is_flag 形式，如 is_active
    • 版本字段：通常为 version
    • 时间戳字段：created_at, updated_at
    • 关联外键：target_entity_id, 如 author_id, group_id

    逻辑设计规范

    1. 范式和反范式

    • 一般遵循第三范式
    • 适当冗余换取性能(反范式设计)，比如物化视图等
    • 优先满足实时 OLAP，其次才是一致性

    2. 主键设计

    • 推荐自增 ID 为主键(序列类型或 serial)
    • 避免 UUID 作为物理主键(可做业务主键)
    • 除非业务要求，不要复合主键

    3. NULL 值处理

    • 适当设置 NOT NULL 约束
    • 尽量提供 DEFAULT 约束
    • 索引可包含 null，但是唯一约束不推荐对可能为空的列建立

    物理设计规范

    数据类型选择

    • 字段长度确定的情况下优先用 char(n) 替代 varchar(n)
    • 大多数情况下用 timestamp 代替 date 和 time
    • 范围确定且较少用到整数时，优先用 smallint 节省空间
    • 不要超过 BIGINT 的范围不要使用 NUMERIC 类型(因为开销巨大)
    • 逻辑开关字段建议用 bool(而非 tinyint)
    • 文本类型：小于 1k 用 VARCHAR，超长文本用 TEXT
    • 精度金额字段用 numeric(precision, scale)，避免浮点数产生误差

    扩展应用规范

    • 用 JSON 数据类型存取变结构数据
    • 用 ltree 类型存储树形数据(如组织机构，分类等)
    • 推荐 PostGIS 作为位置服务解决方案

    分区与索引

    1. 合理使用索引

    • 为常用 WHERE 条件建立索引
    • 为高频 ORDER BY 建立索引
    • 为经常 JOIN 的字段建立索引
    • 覆盖索引：SELECT 的字段全部覆盖
    • 建立组合索引要遵守最左匹配原则

    2. 索引注意事项

    • 谨慎使用索引过多
    • 定期维护和清理无效索引
    • 大数据量更新时，暂时关闭约束和索引可加速导入过程

    3. 选择分区策略

    • 大数据量单表 (>1000万行) 时考虑分区表
    • 按日期分区，如按天、月或年进行 RANGE 分区
    • 按地区或类别等离散值，采用 LIST 分区
    • HASH 分区适用于随机分散查询的场景

    其他 PostgreSQL 专属规范

    1. 连接方式优化

    • 使用 PgBouncer 或 PgPool 做连接池
    • 适当设置 max_connections、shared_buffers 等配置
    • 调优 vacuum、autovacuum 策略

    2. 监控指标关注

    • 连接数使用率、锁等待事件
    • 缓冲区命中率、磁盘 I/O
    • 长事务、慢查询等指标

    MongoDB 最佳实践

    MongoDB 命名规范

    1. 数据库命名规范

    • 使用小写字母和数字，多个单词之间用下划线连接
    • 名称长度建议在3-63个字符之间
    • 避免以系统保留字开头，如 system. 等

    2. 集合(Collection)命名规范

    • 集合名称必须全部为小写字母
    • 多个单词使用下划线分隔，如 user_orders
    • 不使用特殊字符和符号
    • 推荐使用复数形式，如 users, orders 等
    • 长度限制在1-128个字符以内

    3. 字段(Field)命名规范

    • 使用 camelCase 风格的命名方式
    • 简短、明确、描述性
    • 始终使用双引号来表示键
    • 使用嵌套文档和数组来组织相关数据
    • 避免使用 MongoDB 保留的关键字

    数据模型设计

    1. 内嵌与引用的设计

    • 一对少量关系时使用内嵌文档 (Embedded Documents)
    • 一对大量关系时使用引用关系 (References)
    • 变更频繁的子文档应考虑独立设计

    2. 文档大小限制

    • 单个文档限制为16MB，尽量保持小体积
    • 预估未来数据增长，防止文档超出上限
    • 使用 GridFS 存储大容量文件

    3. 索引设计原则

    • 为常用查询字段建立索引
    • 合理设计复合索引，考虑索引前缀原则
    • 考虑查询覆盖，使查询只涉及索引字段
    • 定期分析索引使用情况并清理无用索引

    配置优化

    1. 网络配置

    • 启用认证
    • 使用 TLS/SSL 加密连接
    • 配置防火墙规则
    • 限制 bindIP 只接受必要来源

    2. 存储配置

    • cacheSizeGB: WiredTiger 存储引擎的缓存大小，通常设置为主机内存的一半
    • syncdelay: 系统数据刷新间隔（秒），默认 60 秒
    • blockCompressor: 选择合适的数据压缩算法

    3. 内存与缓存配置

    • 默认缓存大小按总内存一半计算
    • 系统总内存为 8GB 时，默认缓存 3.5GB

    安全配置要点

    1. 网络安全

    • 启用认证并使用强密码策略
    • 使用角色而非特权用户进行日常操作
    • 定期进行权限审查
    • 禁用 bindIpAll 并指定具体的监听 IP

    2. 账户安全

    • 遵循最小权限原则创建用户账户
    • 定期轮换密码
    • 使用专用管理账户
    • 使用 SCRAM 身份验证机制

    3. 角色管理

    • 数据库用户角色: read, readWrite
    • 数据库管理角色: dbAdmin, dbOwner, userAdmin
    • 集群管理角色: clusterAdmin, clusterManager, clusterMonitor, hostManager
    • 备份恢复角色: backup, restore
    • 全局角色: root (最高权限)

    部署与运维最佳实践

    1. 生产环境部署注意事项

    • 定期轮换密钥文件
    • 配置适当的日志级别和归档策略
    • 配置文件需要先验证格式再重启服务
    • 使用 systemd 进行服务管理

    2. 版本升级注意事项

    • 做好充分的数据备份
    • 在测试环境先行验证
    • 按照推荐路径升级，不跨版本过快升级

    3. 备份恢复最佳实践

    • 建立定期备份策略
    • 测试备份的有效性和恢复流程
    • 在不同的存储媒介上保留备份副本
    • 定期进行灾难恢复演练

    4. 监控最佳实践

    • 设置关键指标的告警
    • 定期分析慢查询日志
    • 监控数据库资源使用情况
    • 建立性能基线用于对比分析

    通用数据库规范

    1. 连接管理：使用连接池以避免频繁建立连接，提升性能
    2. 异常处理：做好数据库异常捕获与处理，增强系统健壮性
    3. 事务管理：控制事务边界，避免长事务阻塞，减少锁竞争
    4. SQL语句：避免复杂的多表关联，减少数据库计算压力
    5. 数据一致性：遵循ACID原则确保数据完整性，正确使用锁机制

    集群管理最佳实践

    1. 高可用性：采用主从复制、集群模式提高可用性
    2. 数据分片：合理规划数据分布以应对海量数据挑战
    3. 负载均衡：通过中间件等方式均衡读写请求
    4. 故障切换：设置合理的故障检测与自动切换机制
